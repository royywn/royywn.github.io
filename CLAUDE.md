# CLAUDE.md — Personal Portfolio & Quant Blog

## Project in one paragraph
A static personal site for Roy Yang (data scientist, finance) hosted on GitHub Pages.
Purpose: showcase quantitative trading projects, agentic AI work, and technical writing
to support a job search in data science / quant finance. The site must look professional,
load fast, and make it trivially easy for a recruiter to reach the GitHub repos and CV.

## Source-of-truth documents
- `ARCHITECTURE.md` — stack, directory layout, routing, deployment. Read before structural changes.
- `DESIGN.md` — colors, typography, components, landing copy, voice. Implement literally;
  where it is silent, choose the quieter option.
- `PLAN.md` — phased milestones with verifiable completion criteria. Work top-down; do not skip phases.
- `DECISIONS.md` — append-only ADR log. Add an entry BEFORE implementing any deviation from ARCHITECTURE.md.
- `BLOCKERS.md` — anything requiring human input. Append and move on; never guess on blocker items.

## Tech stack (fixed — see DECISIONS.md before changing)
- Astro 4.x, static output (`output: 'static'`)
- Content collections for `blog/` and `strategies/` (markdown + frontmatter, zod-validated schemas)
- Tailwind CSS for styling; no UI component library
- Pagefind for search (build-time index), giscus for comments, @astrojs/rss for feed
- Deploy: GitHub Actions → GitHub Pages on push to `main`
- Authoring pipeline: Jupyter notebooks in `notebooks/{strategies,blog}/` are the
  source of truth for technical content; `scripts/convert_notebooks.py` (nbconvert)
  renders them into `src/content/` at build time. Notebooks are NEVER executed in
  CI — outputs are published exactly as saved.
- NO server-side code, NO database, NO client-side framework unless a specific
  interactive widget requires it (then: a single Astro island, vanilla JS or Preact)

## Commands
- `npm run convert` — run `scripts/convert_notebooks.py` (ipynb → markdown into src/content/)
- `npm run dev` — local dev server (runs convert first via predev)
- `npm run build` — convert + production build (MUST pass with zero errors before any commit)
- `npm run preview` — serve the built site locally
- `npx astro check` — type/content-schema validation (MUST pass)

## Hard rules
1. Every page must build statically. If you write `fetch()` at runtime, stop and reconsider.
2. Never invent content. Placeholder copy goes in as `<!-- TODO:ROY ... -->` markdown comments
   and gets logged in BLOCKERS.md. Do not write fake backtest results, fake metrics,
   fake testimonials, or biographical claims not present in `content-sources/`.
3. No emoji in headings or body copy. Tone: precise, understated, engineer-to-engineer.
4. All internal links must be root-relative and verified by the link-check step in CI.
5. Frontmatter schemas are contracts: changing one requires a DECISIONS.md entry and
   migration of all existing content files in the same commit.
6. Files generated by the notebook converter carry a `<!-- GENERATED from notebooks/... -->`
   header and are gitignored; NEVER hand-edit generated markdown — fix the notebook
   or the converter. Notebooks are never executed by any script or CI step.
7. Lighthouse budgets (CI-enforced targets): Performance ≥ 95, Accessibility ≥ 95,
   total JS shipped to client ≤ 50 KB on content pages.
8. Commit style: conventional commits (`feat:`, `fix:`, `content:`, `chore:`).
   One logical change per commit. Run build + check before every commit.

## Session loop (per autonomous run)
1. Read PLAN.md → find first unchecked item in the current phase.
2. Read relevant ARCHITECTURE.md section.
3. Implement smallest complete increment. Build + check + (if applicable) link-check.
4. Tick the PLAN.md checkbox with a one-line note and commit hash reference.
5. If blocked → append to BLOCKERS.md with context and what input is needed; pick the
   next unblocked item.
6. Stop conditions: phase complete, or 3 consecutive failed build attempts on the same
   item (log to BLOCKERS.md and stop), or all remaining items blocked.

## Content guardrails (job-search context)
- The site represents a real job candidate. Accuracy > polish > volume.
- Strategy pages must include an honest "Limitations" section (transaction costs,
  overfitting risk, data constraints). Overclaiming is a defect, not a feature.
- Never publish anything from `content-sources/private/` — it is gitignored; verify
  it stays that way after any change to `.gitignore`.
